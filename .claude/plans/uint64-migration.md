# Plan: Change userId and orderId from int64 to uint64

## Rationale
`userId` and `orderId` are generated by snowflake (always positive) or AUTO_INCREMENT (always positive). Making them `uint64` at the type level eliminates the possibility of negative modulo in worker sharding (`bill_executor.go`, `fill_executor.go`) and table sharding functions, preventing potential panics from negative array indices.

## Key Considerations

- **Snowflake**: `bwmarrin/snowflake.ID` is `int64` internally. We cast at `GetId()` boundary: `return uint64(node.Generate().Int64())`. Since snowflake IDs are always positive, this is safe.
- **MySQL**: GORM handles `uint64` transparently. All DB columns remain `BIGINT UNSIGNED` (or become so if they were `BIGINT SIGNED`). This may require a DB migration — but since IDs are never negative, it's a safe change.
- **JSON serialization**: `uint64` serializes as number in JSON. JavaScript's `Number.MAX_SAFE_INTEGER` is 2^53-1, and snowflake IDs can exceed this. However, this is already the case with `int64` snowflake IDs — no regression. The existing `vo.go` already converts IDs to strings via `I64ToA` for JSON responses, so this is handled.
- **strconv**: Replace `strconv.FormatInt(id, 10)` with `strconv.FormatUint(id, 10)` and `strconv.ParseInt(s, 10, 64)` with `strconv.ParseUint(s, 10, 64)`.
